# Shashclaw Development Guide

You are helping develop **Shashclaw** — a self-hosted AI agent platform built with React 19, TypeScript, Express, Drizzle ORM, and PostgreSQL (Neon).

## Core Stack & Rules

- **Frontend**: React 19 + Vite + Tailwind CSS + shadcn UI
- **Backend**: Express + TypeScript + tRPC
- **Database**: Drizzle ORM → Neon PostgreSQL
- **LLM**: Anthropic (primary) with Gemini fallback
- **Deployment**: Railway (watches `develop` branch)
- **Branch**: Always work on `develop`. Never commit to `main`.

### Key Rules

1. Use `grep` first, never read whole files
2. Fix only what's broken — no unrelated changes
3. Run `pnpm build` before every commit
4. Test locally with `pnpm dev` before pushing
5. Never hardcode URLs/keys; use `process.env`
6. Commit after each fix with clear message

## Architecture Overview

### 1. Request Lifecycle

**Client → API → Auth → DB → LLM → Response**

- **tRPC Routes**: All procedures in `server/routers.ts`, mounted at `/api/trpc` via `server/_core/index.ts`
- **REST Auth**: Routes in `server/customAuth.ts` (demo-login, verify-token, logout)
- **Authentication**: `server/_core/sdk.ts` → JWT session cookie + `server/_core/context.ts` for tRPC context
- **Database**: `server/db.ts` → Drizzle → Neon via `DATABASE_URL`
- **LLM**: `server/_core/llm.ts` → Direct Anthropic API (key in `ANTHROPIC_KEY` env var)

### 2. Core Files & Concepts

- **Auth**: `server/customAuth.ts` (no OAuth/Manus dependency; pure JWT + demo-login)
- **DB Access**: `server/db.ts` (getDb, upsertUser, etc.) + `server/integrations.ts` (agent/skill helpers)
- **API Health**: `GET /api/health` and `GET /api/auth/demo-login` are critical smoke tests
- **Environment**: `server/_core/env.ts` loads all config from `process.env` (DATABASE_URL, ANTHROPIC_KEY, etc.)

### 3. Feature Checklists

**Core features that MUST work**:
- ✅ Login (demo + real auth)
- ✅ Dashboard loads
- ✅ Agent creation
- ✅ Chat with LLM

## Common Tasks

### Before Committing

Use the **Verify Before Commit** workflow:

1. Build: `pnpm build`
2. Start dev: `pnpm dev` (wait for "Server running...")
3. Test `/api/health` → expect 200
4. Test `/api/auth/demo-login` → expect 200 or 302
5. Browser check: login, dashboard load, create agent, send chat, check console
6. **Only commit if all pass**

### Debugging Request Failures

Use the **Request Lifecycle Review** skill to trace:

1. **Client → API**: Is the route registered? Is the path correct?
2. **Auth**: Is the session cookie valid? Is `JWT_SECRET` set? Does the user exist in DB?
3. **DB**: Is `DATABASE_URL` set? Can Neon be reached?
4. **LLM**: Is `ANTHROPIC_KEY` set? Does the Anthropic API return 2xx?
5. **Response**: Does the handler return the expected shape?

### Monitoring App Health

Run the **App Health Check** agent to:

1. Start `pnpm dev`
2. Verify `/api/health` responds
3. Verify `/api/auth/demo-login` works
4. Test DB connection via Drizzle
5. Test LLM call to Anthropic
6. Report pass/fail; fix failures before proceeding

## Environment Variables

Set in `.env.local`:

```bash
DATABASE_URL=postgresql://...        # Neon Postgres
ANTHROPIC_KEY=...                    # Anthropic API key (direct)
```

Do NOT commit `.env.local`. Railway injects these at deploy.

## Troubleshooting

| Issue | Diagnosis | Fix |
|-------|-----------|-----|
| 500 from tRPC | Check server logs for errors in procedure handler | Add logging; re-run with `pnpm dev` |
| Auth redirects to login | Session cookie missing or invalid | Hit `/api/auth/demo-login` first; check `JWT_SECRET` |
| "Cannot read property of null" in chat | LLM call failed or returned unexpected shape | Check `ANTHROPIC_KEY` is set and valid |
| DB query hangs | `DATABASE_URL` invalid or Neon unreachable | Verify URL in .env.local; test with psql or Drizzle studio |
| Build fails | TypeScript or bundler error | Check `pnpm check:types` output; fix type errors |
